01 The two-stage pattern

Good retrieval is almost always two stages: a cheap, high-recall candidate generator that scans everything, then an expensive, high-precision reranker that reorders a small shortlist.

You can't run an expensive scorer over a million atoms per turn — but you can run it over the top 50. So stage 1 trades precision for speed and breadth; stage 2 trades breadth for precision on what survived.

MemAtom today does a strong stage 1 (hybrid vector + graph) and a lightweight stage 2 (a weighted linear blend). Everything in this report is about making stage 2 — the reranker — do more of the heavy lifting, where most of the quality gains live.

Mental modelStage 1 decides what's even considered. Stage 2 decides what wins. If a relevant atom never enters the candidate pool, no reranker can save it — so recall@candidate-pool is the ceiling on everything downstream.

02 Stage 1 — candidate generation

MemAtom runs two independent retrievers in parallel and unions their candidates. Each catches what the other misses.

Vector path — semantic similarity

The query is embedded (local MiniLM, 384-d) and matched against the atom index in Postgres + pgvector using an HNSW graph index with cosine distance. We pull the nearest 50.

cosine similaritysimilarity = 1 − cosine_distance(q, d) = (q · d) / (‖q‖‖d‖) ∈ [0, 1]
-- pgvector, HNSW, pre-filtered by scope/status
SELECT id, content, 1 - (embedding <=> $q::vector) AS similarity
FROM atoms
WHERE agent_id = $a AND status = 'warm'
ORDER BY embedding <=> $q::vector   -- nearest first
LIMIT 50;

Bi-encoder retrieval: query and document are embedded separately, then compared by a cheap vector op. Fast (sub-millisecond with HNSW), but the two never "see" each other — there's no token-level interaction. Hold that thought; it's exactly what the cross-encoder in §06 fixes.

Graph path — entity traversal

Entities mentioned in the query are matched against the knowledge graph (entities nodes, atom_entities edges) and traversed 1–2 hops. This catches atoms that share an entity but have near-zero vector similarity — the classic "everyone on Project Titan" query that pure embeddings smear away.

graph proximity (fusion.ts)graphProximity(hop, overlap) = hopWeight · min(1, overlap / 2) where hopWeight = 1.0 if hop ≤ 1 (direct mention) = 0.5 if hop = 2 (shared-entity neighbor)
Why hybrid beats either aloneVector answers "what's semantically near X." Graph answers "what's relationally connected to X." Reciprocal-rank-fusing the two lists gives a measured +10–12% recall@10 over vector-only in classic hybrid-search evals — and it's the reason a graph-only atom (similarity 0) can still surface.

03 The current ranker — weighted hybrid fusion

Today, fuseHybrid() merges the two candidate lists and scores every atom with a weighted linear blend of four normalized signals, then sorts and takes the top-K.

current scoring functionscore(d) = w_v·similarity + w_g·graphScore + w_r·recency + w_p·pinned defaults: w_v = 0.55 w_g = 0.25 w_r = 0.15 w_p = 0.05

Each signal is deliberately kept in [0, 1] so the weights are comparable:

  • similarity — cosine, already 0–1 (0 if the atom only came from the graph path).
  • graphScore — graphProximity(), 0–1 (0 if only from the vector path).
  • recency — exponential decay with a 14-day half-life:
recency decay (fusion.ts)recency(d) = exp( −ln2 · ageDays / halfLifeDays ), halfLife = 14 → 1.0 at age 0, 0.5 at 14 days, 0.25 at 28 days
  • pinned — 1 if the user pinned it, else 0 (a small, honest nudge — pins mostly act earlier, by clamping salience).

Two more rules shape the output:

  • Source tagging — an atom is credited as a vector match only if its similarity clears VECTOR_SOURCE_FLOOR = 0.15; it's a graph match if traversal found it. The sources array makes every result explainable.
  • Noise gate — atoms that matched no meaningful path (low similarity, no graph link) are dropped, not ranked. This is a cheap precision guard.
// fusion.ts — the heart of stage 2 today
for (const e of byId.values()) {
  if (e.sources.size === 0) continue;            // noise gate
  const recency = recencyDecay(e.atom.createdAt, now, halfLife);
  const score =
      w.vector  * e.similarity
    + w.graph   * e.graphScore
    + w.recency * recency
    + w.pinned  * (e.atom.pinned ? 1 : 0);
  ranked.push({ ...e.atom, score, similarity: e.similarity, graphScore: e.graphScore, sources:[...e.sources] });
}
return ranked.sort((a,b) => b.score - a.score).slice(0, topK);
This is a linear "combSUM"Summing weighted, normalized per-retriever scores is a real, respectable fusion method (CombSUM). It's fast, transparent, and tunable. It is also the floor of what's possible — the next sections are the ceiling.

A weighted sum has three structural weaknesses. Naming them tells you exactly which reranker fixes which.

05 Fix #1 — normalization & Reciprocal Rank Fusion

Before you fuse scores from different retrievers, make them comparable. Two standard moves:

Score normalization (CombSUM, but honest)

Rescale each retriever's scores to a common range per query, then sum.

min-max / z-scoreminmax(s) = (s − min) / (max − min) ∈ [0,1] zscore(s) = (s − μ) / σ zero-mean, unit-variance

Min-max is simple but sensitive to outliers; z-score is more robust when score distributions are roughly normal. This alone fixes the "one retriever dominates" problem without touching weights.

Reciprocal Rank Fusion (RRF) — the workhorse

RRF ignores raw scores entirely and fuses by rank position. It's astonishingly robust — no calibration, no tuning — and is the default fusion in Elasticsearch, OpenSearch, and Azure AI Search.

reciprocal rank fusionRRF(d) = Σ_i 1 / (k + rank_i(d)), k ≈ 60 rank_i(d) = position of d in retriever i's list (1-based) weighted variant: RRF(d) = Σ_i w_i / (k + rank_i(d))

An atom ranked #1 by vectors and #3 by graph scores 1/61 + 1/63. Appearing high in both lists compounds; the constant k damps the influence of low ranks. Because it's rank-based, a graph hit with similarity 0 competes fairly with a strong vector hit.

// drop-in: fuse the vector list and graph list by rank
function rrf(lists, k = 60) {
  const score = new Map();
  for (const list of lists)
    list.forEach((d, i) => score.set(d.id, (score.get(d.id) ?? 0) + 1 / (k + i + 1)));
  return [...score].sort((a,b) => b[1] - a[1]);
}

✅ Pros

  • No score calibration needed
  • Robust across very different retrievers
  • One hyperparameter (k), ~zero cost

❌ Cons

  • Throws away score magnitude (a runaway-best doc looks like a normal #1)
  • Still no query–doc interaction
MemAtom fitEasiest real upgrade: keep recency/pin as additive boosts, but fuse the vector and graph candidate lists with weighted RRF instead of summing raw similarity + graphProximity. It directly removes the score-scale weakness from §04 with a few lines in fuseHybrid().

A cross-encoder concatenates the query and a candidate and runs them through a transformer together, so attention sees every query token against every document token. It outputs one relevance score. That joint attention is what lets it catch negation, qualifiers, and subtle mismatches a bi-encoder smears away.

You never run it over the whole store — only the ~50 candidates stage 1 already shortlisted. That's the entire point of two-stage retrieval.

// recall() becomes a clean two-stage pipeline:

Expected impactOn standard IR benchmarks, adding a cross-encoder reranker over a bi-encoder commonly lifts nDCG@10 by 10–30%. For a memory system this shows up as "it stopped surfacing the almost-right atom."

07 Fix #3 — diversity with MMR

Relevance alone can return three paraphrases of the same fact, burning context budget and starving the prompt of breadth. Maximal Marginal Relevance greedily builds the top-K, penalizing each pick by how similar it is to what's already chosen.

maximal marginal relevancenext = argmax_{d ∈ R∖S} [ λ·rel(d, q) − (1−λ)·max_{d′ ∈ S} sim(d, d′) ] R = candidate pool S = already-selected λ ∈ [0,1] λ = 1 → pure relevance λ → 0 → pure diversity (λ ≈ 0.7 typical)

It reuses the embeddings you already have (sim(d, d′) is just cosine between two atoms), so it's nearly free and runs after the reranker on the shortlist.

MemAtom noteYou already dedupe on write (the dedupeThreshold in capture()). MMR is the read-time complement: even with distinct atoms, it ensures the injected K spans different facts/entities rather than piling onto one topic.

08 Fix #4 — learning-to-rank from feedback

Stop hand-tuning weights. Learning-to-rank (LTR) trains a model to order candidates, using features you already compute plus signals you already collect.

Three flavors

Pointwise — regress a relevance score per doc Pairwise — learn d₁ > d₂ (RankNet) Listwise — optimize the whole ordering (LambdaMART)

LambdaMART (gradient-boosted trees, e.g. LightGBM rank:ndcg / XGBoost) is the battle-tested default — it directly optimizes a ranking metric like nDCG and handles heterogeneous features well.

Features — you already have most of these

GroupFeatures
Relevancesimilarity, graphScore, cross-encoder score, hop, entity-overlap count
Memory statesalience, recency, accessCount, pinned, age, atom type
Query/matchquery length, #query entities, lexical overlap (BM25), type-match flag

Labels — for free, from the product

  • Pin → strong positive ("this mattered").
  • Forget / tombstone → strong negative ("wrongly surfaced").
  • Cited in the answer (the LLM used the atom) → positive.
  • Re-ask / follow-up miss → the recalled set was insufficient.
When to reach for LTROnly after you're logging recalls + outcomes and have a labeled set. It's the highest ceiling but the most operationally heavy — and a good cross-encoder often gets you most of the way first.

09 LLM-as-reranker

Hand the candidate shortlist to an LLM and ask it to select/order the most relevant — a listwise reranker with real reasoning. Variants: pointwise ("rate 0–10"), pairwise ("which is more relevant"), and listwise/permutation ("return ids best-first"), the last being strongest.

✅ Pros

  • Best on reasoning-heavy relevance & multi-hop
  • Understands negation, intent, instructions
  • Zero training

❌ Cons

  • Slow + costly per turn — usually too heavy for the hot path
  • Position bias & nondeterminism
  • Context limits on candidate count
Where it shinesUse the LLM reranker offline to generate gold labels (distill it into a cross-encoder or LTR model), or online only for small N on high-value queries. It's a teacher more than a serving-path component.

10 Calibration & abstention

Ranking decides order; calibration decides whether to inject at all. A reranker score of "0.8" should mean "80% likely relevant" — raw model scores rarely do. Map them to probabilities, then set a threshold to abstain when nothing is good enough (the negative/absence case from your eval).

calibrationPlatt: p = σ(a·score + b) (fit a, b on a labeled set) Isotonic: p = monotonic step-fit (non-parametric, more flexible) inject d only if p(d) ≥ τ ; else return ∅ (don't pollute the prompt)
Why it matters for memoryInjecting a confidently-wrong memory is worse than injecting none — it actively misleads the agent. MemAtom's current VECTOR_SOURCE_FLOOR + noise gate is a crude version of this; calibrated thresholds make abstention principled and tunable.

11 A phased upgrade path for MemAtom

Sequenced by effort-to-impact. Each phase ships independently behind the existing interfaces — nothing here is a rewrite.

Phase 1 · Normalize + RRF low effort

Fuse the vector & graph candidate lists with weighted RRF (k=60) inside fuseHybrid(); keep recency/pin as additive boosts. Removes the score-scale weakness. Pure code change, no new deps.

Phase 2 · Cross-encoder rerank highest impact

Add a Reranker interface; widen stage-1 to top-50, cross-encode locally (ms-marco-MiniLM-L-6-v2 via ONNX) to top-8. Cache scores in Redis. The single biggest nDCG jump.

Phase 3 · MMR diversity low effort

Apply MMR (λ≈0.7) over the reranked shortlist using existing embeddings. Stops near-duplicate atoms from crowding the top-K.

Phase 4 · Calibration + abstention medium

Fit Platt/isotonic on logged labels; replace the hard floor with a calibrated threshold τ. Principled "inject nothing" behavior.

Phase 5 · Learning-to-rank data-gated

Once recalls + pin/forget/cited signals are logged, train LambdaMART on the feature set and swap it in as the reranker. Highest ceiling.

12 How to measure the gains

Never ship a reranker on vibes — every phase above must beat the last on a fixed golden set. Track:

MetricWhat it tells you
nDCG@kPrimary — graded relevance and ordering. The number a reranker should move.
Recall@k / Hit@kDid the gold atom make the injected set? (Bounded by stage-1 pool recall.)
MRRHow high the first relevant atom lands.
Abstention / FP rateOn negative probes — does it correctly inject nothing?
p95 latencyReranking must stay inside the 200 ms budget.

Run each phase as an A/B against the previous on the same probe set; gate CI on "no nDCG regression." The vector-vs-graph and rerank-on/off ablations are exactly the experiments that justify each added millisecond.

Bottom lineMemAtom's stage 1 (hybrid vector + graph) is already strong. The fastest path to noticeably better memory is a real stage-2 reranker — start with RRF for fusion sanity, then a local cross-encoder for the big precision jump, then MMR for breadth. Measure every step on nDCG.
