@mematom/memory-sdk

Developer documentation

A drop-in memory SDK that turns your agent's conversations into structured, retrievable memory — per-message memory atoms, local embeddings (no API key), and GraphRAG recall. Framework-agnostic: two calls fit into any agent loop.

Overview #

MemAtom sits beside your LLM. Before generating, it recalls the most relevant memories and you inject them into context. After replying, it captures new memories from the turn — asynchronously, off the hot path. Memory is scoped to an agentId, not a chat thread, so it persists across sessions.

  • Typed atoms with provenance — episode · fact · preference · trait · skill
  • Hybrid recall — vector similarity and knowledge-graph traversal, reranked
  • Local embeddings — all-MiniLM-L6-v2, no key, fully offline
  • Swappable everything — Embedder, LlmExtractor, MemoryRepository are interfaces

Install #

npm i @mematom/memory-sdk

Peer requirements: Node ≥ 20, and a vector-capable store (Postgres + pgvector is included). The default embedder downloads a ~80 MB model on first use, then caches it.

Quickstart #

Wire the SDK once, then make two calls inside whatever agent loop you already have.

import {
  createMemory,
  LocalEmbedder,
  PostgresMemoryRepository,
} from '@mematom/memory-sdk'

const embedder = new LocalEmbedder()       // local MiniLM, no API key
await embedder.init()                    // load model up front (fail fast)

const mem = createMemory({
  embedder,
  extractor,                          // your LlmExtractor (Claude / OpenAI / …)
  repository: new PostgresMemoryRepository(pool),
  agentId: 'assistant',
})

// ── inside YOUR agent loop ─────────────────────────────
const memories = await mem.recall(userMessage)          // RecalledAtom[]
const reply = await llm(buildPrompt(memories, userMessage))
await mem.capture({                                // fire-and-forget after replying
  sessionId, userMessageId, assistantMessageId,
  userText: userMessage,
  assistantText: reply,
})
Tip: run capture() after you've sent the reply (don't await it on the hot path). Extraction makes its own LLM call, so keeping it off the response path means memory never adds latency.

The memory atom #

Every memory is one structured record. Recall returns these enriched with scoring (RecalledAtom).

{
  id: "9f3b…",
  agentId: "assistant",
  type: "preference",              // episode|fact|preference|trait|skill
  content: "User prefers terse, urgency-first replies.",
  entities: ["tone"],             // graph nodes
  evidence: ["msg_a", "msg_b"],   // messages it was distilled from
  salience: 0.5,                  // drives eviction (M2)
  pinned: false,                 // user "tattoo" → never forgotten
  status: "warm",               // warm|cold|tombstoned
  author: { type: "agent", id: "assistant" },
}

Recall & capture #

The two verbs that make up the loop:

  • recall(query) — embed the query, run vector + graph retrieval, rerank, return the top‑K atoms to inject into your prompt.
  • capture(turn) — ask the extractor for atom proposals, embed each, drop near-duplicates, persist the rest (and append an audit op).
Graceful by design: if recall fails, it returns [] rather than throwing — your turn still answers, just without memory. If capture fails, the reply already happened, so nothing user-facing breaks.

GraphRAG retrieval #

Recall runs two independent paths and fuses them, so you catch what pure vector search misses:

  • Vector — semantic similarity over the embedding index (pgvector, cosine).
  • Graph — entities mentioned in the query are matched against a knowledge graph (entities nodes, atom_entities edges) and traversed 1–2 hops.
  • Rerank — the union is scored by a weighted blend of vector · graph · recency · pin.

Each returned atom carries a sources array ("vector" / "graph") so every recall is explainable. Set enableGraph: false for vector-only retrieval.

mem.recall(query, topK?) #

recall(query: string, topK?: number): Promise<RecalledAtom[]>

Embeds the query, runs hybrid retrieval, and returns the top‑K atoms ordered by score (highest first). Never throws on an empty store — returns [].

const hits = await mem.recall("what is the user allergic to?", 5)
hits[0] // → { content, score, similarity, graphScore, sources:["vector","graph"], … }

mem.capture(turn) #

capture(turn: Turn): Promise<CaptureResult> → { created: Atom[], skipped: number }

Extracts atoms from a completed turn, embeds + dedupes them, and persists the survivors. skipped counts near-duplicates dropped.

const { created, skipped } = await mem.capture({
  sessionId, userMessageId, assistantMessageId,
  userText, assistantText,
})

Interfaces (bring your own) #

Everything external is an interface — swap any of them without touching the rest.

Embedder

interface Embedder {
  readonly dim: number
  embed(text: string): Promise<number[]>
  embedBatch(texts: string[]): Promise<number[][]>
}

Default: LocalEmbedder (all-MiniLM-L6-v2, 384‑dim). Call init() at boot to load eagerly.

LlmExtractor

interface LlmExtractor {
  // returns raw JSON: { atoms: [{ type, content, entities }] }
  propose(input: { userText: string; assistantText: string }): Promise<unknown>
}

The SDK validates the returned JSON with Zod, so propose can return anything shaped like { atoms: [...] } — use provider tool-calling / JSON mode.

MemoryRepository

Persistence + retrieval. PostgresMemoryRepository is included; implement the interface for any other store. Key methods:

  • insertAtom, searchAtomsByVector, findSimilarForDedup
  • findEntitiesInText, graphTraverse — the GraphRAG hooks
  • listAtoms, getAtom, recordAccess
  • pinAtom, tombstoneAtom — pin / forget
  • createSession, addMessage, getRecentMessages

Recipes #

Swap the embedder (hosted API)

class ApiEmbedder implements Embedder {
  dim = 1536
  async embed(text) { return callEmbeddingsApi(text) }
  async embedBatch(texts) { return Promise.all(texts.map(this.embed)) }
}
createMemory({ embedder: new ApiEmbedder(), … })

Use any LLM for extraction

const extractor: LlmExtractor = {
  async propose({ userText, assistantText }) {
    // return { atoms: [{ type, content, entities }] } via tool-calling / JSON mode
    return await myLlmJson(userText, assistantText)
  },
}

Vector-only (disable graph)

createMemory({ …, enableGraph: false })

Pin & forget #

Memory is user-correctable. Pinning clamps salience to 1.0 (never evicted); forgetting tombstones the atom and nulls its embedding — both append an audit op.

await mem.repository.pinAtom(atomId, true)        // 📌 keep forever
await mem.repository.tombstoneAtom(atomId, "user asked to forget")

Testing #

Because the dependencies are interfaces, you can test your integration with deterministic fakes — no network, no flake. Inject a fake Embedder (hash → stable vector) and fake LlmExtractor (scripted JSON), and assert on recall ordering and captured atoms. The SDK ships 41 tests built exactly this way, plus real pgvector integration tests.
